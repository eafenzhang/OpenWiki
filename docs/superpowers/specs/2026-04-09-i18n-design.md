# Xiaoyun i18n Design — Chinese/English Bilingual Support

## Goal

Add Chinese/English language switching to Xiaoyun so the open-source release is accessible to English-speaking users. Language auto-detects from the OS and can be overridden in Settings.

## Scope

- **In scope**: All frontend UI text (labels, buttons, placeholders, error messages, status text, tooltips, relative time expressions)
- **Out of scope**: AI prompts in Rust backend (`prompts.rs`, `wiki_prompts.rs`) — these stay in Chinese. AI model output language is unaffected.

## Architecture

### Library

**react-i18next** + **i18next** + **i18next-browser-languagedetector**

- `i18next`: core i18n engine
- `react-i18next`: React bindings (useTranslation hook, Trans component)
- `i18next-browser-languagedetector`: auto-detect OS/browser language on first launch

### i18n Initialization

New file: `src/lib/i18n.ts`

```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector';

// Import all translation files statically (small app, no lazy loading needed)
import zhCommon from '../locales/zh-CN/common.json';
import zhSettings from '../locales/zh-CN/settings.json';
import zhContent from '../locales/zh-CN/content.json';
import zhWiki from '../locales/zh-CN/wiki.json';
import zhDigest from '../locales/zh-CN/digest.json';
import zhDatahub from '../locales/zh-CN/datahub.json';

import enCommon from '../locales/en/common.json';
import enSettings from '../locales/en/settings.json';
import enContent from '../locales/en/content.json';
import enWiki from '../locales/en/wiki.json';
import enDigest from '../locales/en/digest.json';
import enDatahub from '../locales/en/datahub.json';

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    resources: {
      'zh-CN': {
        common: zhCommon,
        settings: zhSettings,
        content: zhContent,
        wiki: zhWiki,
        digest: zhDigest,
        datahub: zhDatahub,
      },
      en: {
        common: enCommon,
        settings: enSettings,
        content: enContent,
        wiki: enWiki,
        digest: enDigest,
        datahub: enDatahub,
      },
    },
    fallbackLng: 'zh-CN',
    defaultNS: 'common',
    interpolation: { escapeValue: false }, // React handles XSS
    detection: {
      order: ['localStorage', 'navigator'],
      lookupLocalStorage: 'xiaoyun-language',
      caches: ['localStorage'],
    },
  });

export default i18n;
```

Import in `src/main.tsx` before app render:

```typescript
import './lib/i18n'; // must be before App import
```

### Language Persistence

**Storage**: `localStorage` key `xiaoyun-language`

- On first launch: `i18next-browser-languagedetector` reads OS language → maps `zh*` to `zh-CN`, everything else to `en`
- When user changes language in Settings: call `i18n.changeLanguage(lng)` → detector auto-saves to localStorage
- No need to touch the Rust/SQLite settings layer — localStorage is sufficient for a UI-only preference and survives app restarts in Tauri's WebView

### Settings UI — Language Switcher

Add a "语言 / Language" option as the **first item in the Appearance (外观) category**, since language and theme are both appearance-level preferences.

Design:

```
外观
  语言 / Language     [  中文  |  English  ]    ← segmented control, same style as theme picker
  主题模式            [  浅色  |  深色  | 跟随系统  ]
```

The language label is always shown bilingually ("语言 / Language") so English users can find it even if the UI is currently in Chinese.

Implementation:

```tsx
const { i18n } = useTranslation();

// Language switcher
<SegmentedControl
  value={i18n.language.startsWith('zh') ? 'zh-CN' : 'en'}
  onChange={(lng) => i18n.changeLanguage(lng)}
  options={[
    { value: 'zh-CN', label: '中文' },
    { value: 'en', label: 'English' },
  ]}
/>
```

### Component Migration Pattern

Each component replaces hardcoded Chinese text with `t()` calls:

**Before:**
```tsx
const TABS = [
  { id: "content", label: "内容", icon: ClipboardList },
  { id: "wiki", label: "知识", icon: BookOpen },
];
```

**After:**
```tsx
const { t } = useTranslation();

const TABS = [
  { id: "content", label: t('nav.content'), icon: ClipboardList },
  { id: "wiki", label: t('nav.wiki'), icon: BookOpen },
];
```

For components with namespace-specific text:

```tsx
const { t } = useTranslation('settings');
// t('appearance.theme') → "主题模式" or "Theme"
```

### Data-Driven Labels (settingsStore)

Some Chinese text lives in store configuration objects (e.g., `PROVIDER_LABELS`, `MODELS_BY_PROVIDER` group names). These must move to translation files:

**settingsStore.ts `PROVIDER_LABELS`:**
```json
// zh-CN/settings.json
{ "provider.dashscope": "阿里云百炼" }

// en/settings.json
{ "provider.dashscope": "Alibaba DashScope" }
```

**Model group names** (e.g., "免费推荐", "更多免费", "智谱"):
```json
// zh-CN/settings.json
{ "modelGroup.freeRecommended": "免费推荐", "modelGroup.moreFree": "更多免费" }

// en/settings.json
{ "modelGroup.freeRecommended": "Free (Recommended)", "modelGroup.moreFree": "More Free" }
```

The store keeps the raw data; components use `t()` to render the display label.

### Brand Name

Brand name is "OpenWiki" in both languages. The header shows "OpenWiki" regardless of language setting. This replaces the previous "小云" branding as part of the open-source release.

### html lang Attribute

Update `<html lang>` reactively when language changes:

```typescript
// in i18n.ts, after init
i18n.on('languageChanged', (lng) => {
  document.documentElement.lang = lng;
});
```

### Tauri System Tray Menu

The tray menu is defined in Rust (`lib.rs` or similar). Tray menu items like "显示主窗口", "退出" need bilingual support. Approach:

- On app startup and language change, the frontend sends the current language to the backend via a Tauri command
- The Rust side stores the language preference and rebuilds the tray menu with the correct language
- This is a **separate, optional phase** — tray menu can stay Chinese initially and be translated in a follow-up

### Floating Bubble (BubbleView)

The bubble window (`BubbleView.tsx`) is a separate Tauri WebView window. It imports from the same `src/` codebase, so it shares the same i18n setup. The `localStorage` is shared within the same Tauri app, so language preference carries over automatically.

## Migration Strategy

Work module-by-module, each step produces a compilable, functional app:

1. **Foundation** — Install deps, create `i18n.ts`, create all 12 translation JSON files (Chinese first, keys only), import in `main.tsx`
2. **Common + App shell** — Migrate `App.tsx` (nav tabs, search, brand), populate `common.json`
3. **Settings** — Add language switcher, migrate all settings text to `settings.json`
4. **Content module** — Migrate content list, cards, bubble
5. **Wiki module** — Migrate all wiki components
6. **Digest module** — Migrate insight/radar components
7. **DataHub + Report** — Migrate data hub and weekly report components
8. **English translations** — Fill all `en/*.json` files with English text
9. **Polish** — Test switching, verify no missing keys, check layout doesn't break with longer English text

## Edge Cases

- **English text is often longer than Chinese** (e.g., "设置" → "Settings", "内容捕获" → "Content Capture"). UI must accommodate with flexible layouts — avoid fixed-width containers for text. Current Tailwind layout uses `flex` and `truncate` which handles this well.
- **Number formatting**: Chinese "3天" vs English "3 days" — handled by i18next interpolation `{{count}}`.
- **Segmented controls**: Buttons with text like "浅色|深色|跟随系统" may need slightly wider min-width for English "Light|Dark|System". Minor CSS adjustment.
- **Missing translation fallback**: `fallbackLng: 'zh-CN'` ensures any untranslated key shows Chinese rather than a raw key string.

## Testing

- Switch language in Settings → all visible text updates immediately, no reload needed
- Close and reopen app → language preference persists
- First launch on English OS → UI shows English
- First launch on Chinese OS → UI shows Chinese
- No console warnings about missing translation keys
- Layout does not overflow or break in either language
